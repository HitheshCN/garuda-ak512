1  System Overview & Architecture

The Garuda AKESC firmware drives a 3-phase brushless motor with no position sensor. Rotor position is inferred from the back-EMF (BEMF) of whichever phase is momentarily not driven. The firmware energises two of the three phases (one high-side PWM, one low-side ground return) and leaves the third floating; the floating phase's terminal voltage crosses the motor's virtual neutral exactly half-way between commutations, and that zero-crossing (ZC) is the timing reference for the entire control loop.

Two ZC detectors run cooperatively: a low-speed software ZC that thresholds the floating-phase ADC sample inside the control ISR, and a high-speed hardware ZC (HWZC) built on the on-chip ADC digital comparators plus SCCP capture/timers. A sector-PI PLL tracks the commutation period in 10 ns resolution and schedules the next commutation autonomously, with timing advance, an RC-filter-lag compensator, and an absolute no-load floor guarding against false harmonic lock.

2.2  Align → open-loop ramp → morph → closed-loop

The bring-up is the firmware's most intricate sequence. In the active sine-startup build:

  • ALIGN — STARTUP_SineInit() seeds the sine angle to 90° (Phase-A peak), amplitude RT_SINE_MIN_AMPLITUDE, releases all PWM overrides (3-phase complementary), and holds for 500 ms while STARTUP_SineAlign() counts alignCounter down.
  • OL_RAMP — STARTUP_SineRamp() integrates an eRPM in Q16 (erpmFrac += RT_SINE_ERPM_RAMP_RATE_Q16), sets the angle increment, and linearly interpolates the amplitude from min to max — a constant-V/f forced field. Commutation here is implicit: the rotating sine field is the forced commutation, looked up from a 256-entry Q15 sineTable256.
  • MORPH — three sub-phases:
    1. CONVERGE blends duty = sine·(256−α)/256 + trap·α/256 with α stepping 0→256 over MORPH_CONVERGE_SECTORS=12 sectors. The trap floating phase uses a virtual neutral (trapDuty+MIN_DUTY)/2, not 50 % — a 50 %-centred float against the ~5–7 % active rails would push ~5 V across 0.13 Ω = 39 A on low-resistance motors; the virtual neutral keeps the disconnection current ≈ 1 A.
    2. WINDOWED_HIZ opens a progressively wider Hi-Z window on the floating phase, polling BEMF inside it while still force-commutating, and adapts timing.stepPeriod toward the measured ZC interval (IIR ≫3).
    3. HIZ full Hi-Z; on goodZcCount ≥ 4 with both rising and falling ZC working and a fresh ZC this tick, zcSynced=true and the state advances to CLOSED_LOOP — a bumpless "hot handoff" that skips ZC re-init and raises the CMP3 threshold to operational.

At the CONVERGE→WINDOWED_HIZ boundary the firmware deliberately drops duty=MIN_DUTY (coast) so near-zero voltage is applied while clean BEMF is acquired, and seeds the forced timing handoff = TIMER1_TO_ADC_TICKS(rampStepPeriod) (see §10 for the historical ×12/5 bug in that macro).

2.4  Fault handling & recovery

Active detectors (all but PCI run in the ADC ISR):

  • OV — vbusRaw > RT_VBUS_OVERVOLTAGE_ADC (3600 ≈ 67 V) for 3 samples → reset CMP3, disable outputs.
  • UV — vbusRaw < uvThreshold (500 ≈ 9.3 V run; relaxed to 400 pre-sync) for 3 samples.
  • OC software — ibusRaw > OC_FAULT_ADC_VAL debounced 3 consecutive ticks (a single rail-spiking sample used to latch a phantom).
  • Board PCI — hardware FPCI (combined OC+OV) in _PWMInterrupt; captures a frozen phase-current snapshot.
  • Desync — two watchdogs: software-ZC timeout, and an HWZC no-capture watchdog (totalZcCount stalls for 150 ms while the autonomous timer keeps commutating on a stale period).

On desync the bridge is disabled and the state → RECOVERY with recoveryCounter = RT_DESYNC_COAST_COUNTS (200 ms coast). After the rotor coasts, if attempts < 3 the firmware re-runs STARTUP_Init() and re-enters ALIGN; a successful sync zeroes the attempt counter. FAULT is terminal until cleared.

3  PWM Generation & 6-Step Commutation

Three PWM generators drive the three half-bridges: PG1 = Phase A (master), PG2 = B, PG3 = C (slaves). All are MODSEL=4 center-aligned, complementary (PMOD=0) with hardware deadtime, and PG2/PG3 are phase-locked to PG1's start-of-cycle. The master holds MPER = LOOPTIME_TCY; MAX_DUTY = LOOPTIME_TCY·99/100.

3.1  Override-based 6-step drive

6-step does not modulate three phases. Each phase is in one of three states per sector, applied through the override registers (PGxIOCON2), written as single atomic 32-bit stores so inter-phase skew is ≈ 30 ns (vs ≈ 90 ns for read-modify-write bitfields — skew on a sector edge can spike Vbus·di/dt past the board fault threshold):

PWM_ACTIVE0x00000000Complementary PWM from PGxDC (HS modulated, LS complementary + deadtime)
LOW0x00301000H forced LOW, L forced HIGH → phase hard-tied to ground (return path)
FLOAT0x00300000Both H and L LOW → half-bridge high-Z (the BEMF-sensed floating phase)

HAL_PWM_SetCommutationStep(step) (hal_pwm.c:1010) indexes the commutation table and writes the three IOCON2 words. HAL_PWM_SetDutyCycle(duty) writes the slaves first, master last (the master write broadcasts an UPDATE so all three load synchronously); only the PWM_ACTIVE phase's duty produces output.

3.2  ADC triggers from the PWM

Only PG1 emits ADC triggers. PG1TRIGA (CAHALF=0, TRIGA=ADC_SAMPLING_POINT) drives ADTR1, the primary BEMF/Vbus/Ibus sample. On the MC510 ADC_SAMPLING_POINT = LOOPTIME_TCY/2 — mid-period in the up-count half = the freewheel / OFF-centre, chosen by an empirical fire-time sweep (MPER/2 reached 215 k eRPM vs 174 k + OC at pulse-centre). A second trigger PG1TRIGB (CAHALF=1, TRIGB=0 = period-centre, mid-ON) is used only by the bus-current experiments.

4  Zero-Crossing Detection

This is the heart of the sensorless scheme. With one phase floating, its terminal voltage swings about the motor's virtual neutral and crosses it half-way between commutations. The firmware compares the floating-phase voltage against a computed neutral threshold; the crossing instant, stamped to 10 ns, is the timing reference the PLL locks to.

4.2  Neutral / threshold mathematics

During PWM-ON the floating terminal sits at roughly half the applied phase voltage, i.e. Vdc·D/2. The threshold is recomputed every ISR tick from the bus ADC and the commanded duty:

rawThresh = (vbusRaw × duty) / ZC_DUTY_DIVISORgaruda_service.c:1040+

where the divisor selects the neutral model:

ZC_DUTY_DIVISOR = 2·LOOPTIME_TCY   →  exact Vdc·D/2   (default, 2810 et al.)
ZC_DUTY_DIVISOR = (26·LOOPTIME_TCY)/10  →  Vdc·D/2.6  (≈ 23 % lower, micros 6/7/8) garuda_calc_params.h:189–192

Because duty/LOOPTIME_TCY = D and vbusRaw is in ADC counts, rawThresh comes out in ADC counts. High-KV micros use the 2.6 divisor to pull the threshold down onto their tiny BEMF neutral. In closed loop the raw value is smoothed by a symmetric ¼-gain IIR (re-seeded instantly on CL entry):

zcThreshSmooth += (rawThresh − zcThreshSmooth + 2) >> 2   (α = ¼, τ ≈ 0.17 ms)garuda_service.c:1124

Optional refinements: FEATURE_VIRTUAL_NEUTRAL replaces the duty model with the measured average (VA+VB+VC)/3; FEATURE_CL_DIFF_IDLE lifts thrDuty by 3·MIN_DUTY for differential idle; and HWZC_THRESH_BIAS_DOWN (=0 by default) can subtract a fixed bias for very-low-BEMF motors. A running peak bemf.bemfDevPeak = max|float − neutral| per sector is captured for the filter compensator (§6).

4.3  Sample point in the PWM cycle

Two distinct instants are sampled (Figure 4): the HWZC comparator channels convert at 1 MHz during ON-time (TRG1SRC=34 = SCCP3), so the digital comparator continuously re-evaluates the ON-time crossing; the 45 kHz ISR bemfRaw is the OFF-centre freewheel sample (TRG1SRC=4 = PG1TRIGA), used by software ZC, the falling-SW detector, and bemfDevPeak. After commutation, detection is masked through a blanking window to skip diode-freewheel/demag ringing:

HW: blankTicks = period × HWZC_BLANKING_PERCENT/100  (14 %), min 1 µs
SW: blankBase = stepPeriod × ZC_BLANKING_PERCENT/100  (3 %); + demag extra when D > zcDemagDutyThresh, capped at stepPeriod/4 hwzc.c:286 · bemf_zc.c

4.4  Falling-SW per-polarity detection

During PWM-ON the driven phase capacitively couples into the floating phase and biases it up. This assists rising-ZC detection but swamps the falling crossing, so on odd (falling) sectors the ON-time comparator goes silent at speed (100 % of misses were on odd sectors). Clean falling BEMF only exists in the PWM-OFF freewheel — exactly the bemfRaw OFF-centre sample.

With FEATURE_HWZC_FALLING_SW=1 the firmware arms the HW comparator only on rising sectors (hwzc.c:341–350); falling sectors are detected by a software compare on the OFF-centre sample:

// falling-SW: in WATCHING, on a falling sector, OFF-centre sample vfo
fth = HWZC_ApplyFilterComp(thresh, falling) - deadband;
if (vfo < fth && intoSector > timerPeriod>>2 && !captureValid) {
    lastCaptureHR = zc; captureValid = true;   // one accept per sector
    goodZcCount++; totalZcCount++;
}

A speed gate HWZC_FALLING_SW_MAX_ERPM (per profile: 35 k A2212, 45 k VEX, 50 k 1407, 70 k 2810) disengages the falling-SW path above it — at high eRPM the OFF-centre sample (~1 PWM period resolution) is too coarse and the rising comparator can carry both polarities. A mirror FEATURE_HWZC_LOWSPD_OFFCTR detects both polarities on the OFF-centre sample at low speed, removing the ON-time min-duty floor and enabling a low-duty CL handoff.

4.5  Validation & rejection logic

HWZC_OnZcDetected() (hwzc.c:402) runs cheap gates first:

  1. First-ZC bypass — the very first ZC after enable skips the plausibility gate and IIR (the seeded lastZcStamp is a half-step guess; the first interval can be 10–190 % of a step). It just anchors the stamp.
  2. PWM-state gate (FEATURE_HWZC_PWM_GATE) — reads the H-side GPIO of the PWMing phase; if PWM is OFF, reject. ON-time crossings (≈ D·Vbus/2) and OFF-time pseudo-crossings (≈ 0 V) are different regimes; mixing them gives 60° slip → 22 A spike.
  3. Interval / plausibility gate — reject if the interval is implausibly short:
    interval < max( stepPeriodHR × HWZC_MIN_INTERVAL_PCT/100 , RT_HWZC_MIN_STEP_TICKS ) → rejecthwzc.c:501
    HWZC_MIN_INTERVAL_PCT = 50. A shorter interval implies the motor > doubled speed in one sector — a PWM-ripple re-fire.
  4. Deadband / hysteresis — the comparator CMPLO is shifted toward the signal by HWZC_CMP_DEADBAND (2 counts for micros, else 4): rising thresh+DB, falling thresh−DB, so ripple straddling neutral doesn't chatter.
  5. Coherence verify (FEATURE_HWZC_VERIFY_READS) — wait one ADC conversion (~1 µs) then re-read a fresh sample. A real crossing stays on the crossed side; a ripple phantom flips back → reject. (Re-reading immediately would only see the trigger sample, hence wait-first.)
What "reject %" (rej) means. Rejections fire mostly during PWM-OFF, so rej% ≈ (1 − duty) — the OFF-window fraction. A high rej% is not guessing: it is the comparator correctly firing on filtered freewheel ripple and being rejected. The meaningful signal is accept-vs-miss per sector (dbgPiMissBySector[6] proved the falling-polarity masking).

On accept the IIR updates only when the ZC is consecutive (stepsSinceLastHwZc==1) and goodZcCount ≥ HWZC_IIR_FREEZE_ZC_COUNT (3 — the freeze prevents a mistimed morph-exit ZC from ratcheting the period to floor).

5  Commutation Timing: PLL, Sector-PI & IIR

The commutation period is tracked in 10 ns SCCP2 ticks. Conversion to/from electrical RPM folds the 100 MHz clock, 6 sectors/erev and 60 s/min into a single constant:

HWZC_TICKS_TO_ERPM(t) = 1×109 / t   |   HWZC_ERPM_TO_TICKS(e) = 1×109 / egaruda_calc_params.h:417

So a stepPeriodHR of 5000 ticks (50 µs) = 200 000 eRPM. The build runs FEATURE_HWZC_SECTOR_PI=1, so the autonomous-timer + PI architecture is live; the reactive IIR path runs only when PI is compiled off, and as the seed.

5.1  The autonomous-timer PLL

In sector-PI mode, detection is decoupled from commutation. SCCP1 (a 32-bit one-shot) is armed in HWZC_OnBlankingExpired to fire at lastCommStamp + timerPeriod; that expiry dispatches to HWZC_OnPiPeriodExpired, which runs the PI correction and then commutates. A ZC arriving between merely timestamps lastCaptureHR — it does not reschedule. A missed ZC simply leaves timerPeriod unchanged (robust by design). This structure lifted the ceiling from ~204 k (reactive wobble) to 240 k+.

5.2  The sector PI controller

The controlled quantity is the position of the captured ZC within the sector. With the autonomous timer firing OnPiPeriodExpired just before commutation, the elapsed time from the sector's start commutation to the ZC is

capValue = lastCaptureHR − lastCommStamp   (modular u32)hwzc.c:828

The setpoint is the expected ZC position, advanced by the schedule (Q8 fixed-point):

advFp8 = ((30 + advDeg)·256)/60  ∈ [128, 256]   |   setValue = (advFp8·T) ≫ 8  ≈ ((30+advDeg)/60)·Thwzc.c:804,829
delta = capValue − setValue   (signed phase error; → 0 at lock)hwzc.c:849

The error is clamped asymmetrically and RPM-dependently before the loop:

T < HWZC_PI_CLAMP_HIGH_TICKS (≥150 k eRPM):   +cap = T/16 ,  −cap = T/32
else (low/mid RPM):   symmetric ±T/8 hwzc.c:851–859

A negative delta means "shrink the period" (commutate faster). At the BEMF ceiling the rotor cannot accelerate, so a negative drift would commutate ahead of the rotor → wrong-angle → 22 A spike → UV cascade. The tighter negative clamp (T/32) blocks that while the looser positive clamp (T/16) keeps the loop responsive to genuine slowdowns. A separate FEATURE_HWZC_HANDOFF_DAMP hard-limits negative delta to T≫6 for the first 24 commutations after CL entry.

The active loop is a parallel-form float PI, gains specified in PLL form (ωn=0.25, ζ=0.5):

integratorF += Ki·delta ,   Ki = ωn2 = 0.0625
T = integratorF + Kp·delta ,   Kp = 2ζωn = 0.25 ,  both clamped ≥ RT_HWZC_MIN_STEP_TICKS hwzc.c:890–898

Because ωn is in rad per sector-event, the loop's absolute bandwidth scales with rotor speed (≈ 56 Hz at 14 k eRPM, ≈ 915 Hz at 230 k) — it widens with speed, matching the rotor timescale. The defensive sub-mode (FEATURE_HWZC_PI_DEFENSIVE) tracks silence: after 6 silent sectors it walks the integrator larger by 1 %/event so a slipping rotor can recapture, exiting after 2 good captures — a soft self-heal that replaced a hard desync watchdog.

5.3  The IIR period filter (reactive seed)

In the reactive path, the period is a first-order IIR over measured intervals:

newPeriod = (3·stepPeriodHR + interval) / 4   (α = ¼) , floored at RT_HWZC_MIN_STEP_TICKShwzc.c:608

guarded by: (1) consecutive-only (stepsSinceLastHwZc==1, so multi-step timeout intervals don't contaminate it); (2) freeze for the first 3 good ZCs after enable; (3) a seqlock (writeSeq++ … writeSeq++) so a lower-priority reader never tears the 32-bit write. In PI mode this IIR is paused — stepPeriodHR is mirrored from the PI's timerPeriod, and a separate slow IIR (shift 11, ≈ 11 ms) feeds the filter compensator.

6  RC Filter-Lag Compensation — Full Mathematics

The board's BEMF sense pin sits behind a passive RC low-pass (τ ≈ 30 µs). A first-order low-pass delays the apparent zero-crossing by a phase angle that grows with rotor speed, so the comparator fires late. The compensator shifts the comparator threshold away from neutral by exactly that lag so the comparator trips early, recovering the true pre-filter ZC. This section gives the complete math, including the ZC threshold-bias term.

6.1  The physical lag

A first-order low-pass with time constant τ introduces, at electrical frequency ωe, a phase lag φ = arctan(ωeτ) ≈ ωeτ for small angles. The firmware works in the small-angle regime and caps the term (below). ωe follows directly from the sector period: with stepPeriodHR = 1e9/eRPM (HR ticks) and felec = eRPM/60, the product ωeτ is computed as a Q15 fixed-point quantity.

6.2  The threshold-bias formula

The compensator HWZC_ApplyFilterComp(thresh, risingZc) (hwzc.h:48) computes the bias step by step:

(1) omegaTauQ15 = HWZC_FILTER_K_Q15 / stepP
(2) if omegaTauQ15 > HWZC_FILTER_MAX_OMEGA_Q15 (24000) → clamp to 24000
(3) amp = bemf.zcAmpForFilterComp ;  if amp==0 → amp = thresh
(4) ampPct = risingZc ? HWZC_FILTER_AMP_PCT : HWZC_FILTER_AMP_PCT_FALLING  (both 50)
(5) offset = ((amp × omegaTauQ15) ≫ 15) × ampPct / 100
(6) if offset > HWZC_FILTER_MAX_OFFSET (600) → clamp to 600
(7) rising: return thresh − offset   |   falling: return thresh + offset hwzc.h:70–98

The constant HWZC_FILTER_K_Q15 folds τ, the SCCP clock and the Q15 scale so that step (1) yields ωeτ in Q15 directly:

HWZC_FILTER_K_Q15 = 2π · 109 · τ · 32768 / 60  =  2π · 109 · 30×10−6 · 32768 / 60  ≈  102 943 706 garuda_config.h:1236

Why this works: stepP (HR ticks) = 109/eRPM and felec = eRPM/60, so

omegaTauQ15 = K/stepP = [2π·109·τ·32768/60] · [eRPM/109] = 2π·(eRPM/60)·τ·32768 = (ωeτ)·32768

i.e. the division by stepP turns the folded constant into ωeτ expressed in Q15 (32768 = 1.0 rad). The cap of 24000 limits ωeτ to ≈ 0.73 rad (≈ 42°), preventing the offset blowing up at very high RPM (the cap engages around 200 k).

The amplitude scaling (step 5) converts the phase offset into an ADC-count threshold shift. For a near-sinusoidal BEMF of peak amplitude amp crossing zero, a small time/phase shift δφ moves the crossing voltage by ≈ amp·δφ; here δφ = ωeτ (Q15), so (amp × omegaTauQ15) ≫ 15 = amp·ωeτ is the count-shift needed to trip the comparator exactly one filter-lag early. The × ampPct/100 (50 %) is an empirical de-rating — the linear small-angle model over-predicts, and 50 % matched the bench at the proven speeds.

6.3  The ZC threshold-bias chain end-to-end

It is worth following the full threshold a comparator actually sees, because three distinct biases stack:

CMPLOfinal = [ zcThreshold ] ∓ offsetfiltComp ± HWZC_CMP_DEADBAND
  • zcThreshold — the duty-model neutral vbusRaw·duty/ZC_DUTY_DIVISOR, ¼-IIR-smoothed (§4.2).
  • ∓ offsetfiltComp — pulls the threshold toward the oncoming signal (down for rising, up for falling) by the filter-lag amount above, so the comparator fires early enough to cancel the RC delay.
  • ± deadband — pushes the threshold a fixed 2–4 counts away from neutral in the crossing direction (rising +DB, falling −DB) so PWM ripple straddling neutral cannot chatter the comparator.

Sign convention together: for a rising sector the net is thresh − offset + DB; for a falling sector thresh + offset − DB. The two biases act in opposite senses by design — the filter-comp advances the trip, the deadband guards against chatter — and the same HWZC_ApplyFilterComp result is used at every CMPLO write site (initial arm in HWZC_OnCommutation, the live ISR refresh, the verify path, and the falling-SW software compare) so all paths stay consistent.

6.4  The amplitude source — why measured, not duty-derived

The offset uses zcAmpForFilterComp — the measured BEMF amplitude (a slow ≈ 46 ms IIR of bemfDevPeak) — not the live zcThreshold. This is critical: zcThreshold ∝ duty, so feeding it into the offset would make the compensation duty-dependent and create a regen positive-feedback path (more duty → bigger offset → earlier fire → period shrinks → more duty). The measured amplitude lags with rotor inertia and reflects the real BEMF magnitude the RC lag acts on. This is the "bemfDevPeak vs zcThreshold" fix that centred the ZC from ~15 % to ~50 % of the sector.

A second positive-feedback break is in the period source: in PI mode stepP for step (1) is the slow IIR stepPeriodForFilterComp (shift 11), not the live timerPeriod. Otherwise a PI period-shrink would grow ωeτ → grow the offset → fire earlier → shrink the period further. The slow IIR keeps that loop gain ≪ 1.

7  Timing Advance

This is 6-step commutation phase advance, NOT field weakening. There is no d-axis current and no space-vector modulation. Advance simply moves the commutation instant earlier in the sector to compensate the inductive lag (phase current cannot switch instantly) plus the detection/filter latency, which lets the motor run past its nominal no-load speed.

The advance angle is a linear interpolation of speed, identical in all three sites (ComputeCommDelay, HWZC_OnPiPeriodExpired, the SW path):

eRPM ≤ RT_RAMP_TARGET_ERPM  →  advDeg = TIMING_ADVANCE_MIN_DEG (0°)
eRPM ≥ RT_TIMING_ADV_FULL_ERPM  →  advDeg = RT_TIMING_ADV_MAX_DEG
else  →  advDeg = MIN + (MAX−MIN)·(eRPM − RAMP_TARGET)/(FULL − RAMP_TARGET) hwzc.c:307–323

Advance is baked into both sides of the loop: in reactive mode the schedule fires early — commDelay = stepPeriodHR·(30−advDeg)/60 — and in PI mode the setpoint shifts — setValue = ((30+advDeg)·T)/60 — so the loop holds the same phase relationship. The endpoint RT_TIMING_ADV_FULL_ERPM is decoupled from the speed cap (it defaults to RT_MAX_CLOSED_LOOP_ERPM but can be overridden), so a motor operating far below its cap still gets meaningful advance growth instead of a near-flat ramp.

8  ABS_FLOOR False-Lock Guard & the λ Math

The relative gates of §4.5 are ratiometric; they cannot catch a cumulative harmonic ratchet where the PI slowly walks onto a half- or third-period phantom lock. FEATURE_HWZC_ABS_FLOOR adds an absolute operating-point floor from no-load physics: the rotor cannot spin faster than the no-load speed for the present duty and bus voltage.

The no-load sector period in HR ticks is computed from the motor's flux constant λ (= focKeUvSRad, in µV·s/rad):

dutyFrac = duty / LOOPTIME_TCY  ;  vbus_v = vbusRaw × VBUS_SCALE_V_PER_COUNT  ;  λ = gspParams.focKeUvSRad
Pff = 181.380 × λ / (vbus_v × dutyFrac)   [HR ticks]
floorP = Pff × (100 / HWZC_ABS_FLOOR_OVERSPEED_PCT)   (overspeed 130 → allow 130 % of no-load) hwzc.c:920–953

The constant 181.380 folds the SCCP clock (108), the 6 sectors/erev, the 2π, and the BEMF↔period algebra so Pff comes out directly in 10 ns ticks. The guard is applied only above HWZC_ABS_FLOOR_MIN_DUTYFRAC=0.03 and vbus_v > 6.

SHRINK-ONLY: clamp only when (timerPeriod < floorP) AND (timerPeriod < T) — the PI is driving the period below the floor.hwzc.c:942

A period already below the floor but growing is a genuine high-speed coast-down through the floor and is left alone (the rotor really is that fast); only an accelerating shrink at idle duty is provably a false-lock.

Why a wrong λ pins the motor. Pff ∝ λ. If λ is too high (e.g. the 2810's 583 used on a motor whose true flux is 230), Pff and floorP inflate, the floor sits above the real no-load period, and the guard clamps the period up to floorP every sector — pinning eRPM at 130 %·(λtrue/λset) = 130 %·(230/583) ≈ 51 % of no-load with large reactive current. This is exactly the VEX "half-speed/high-current" symptom; the fix was setting focKeUvSRad = 5.5132×106/(KV·PP). The same formula seeds the PI integrator at handoff (one-shot, slow-only).

9.4  Outer speed PI (parked)

FEATURE_SPEED_PI=0. A per-ZC eRPM control loop exists (speed_pi.c) working in eRPM space (constant plant gain ≈ 3.2 eRPM/tick vs 16× variation in period space). Algorithm per ZC: throttle→target map, measured = 1e9/stepPeriodHR, a 5 % deadband (error shifted not clamped), a bench-fit feedforward ff = 0.310·target + 400, a PI on the residual (Kp=0.03, Ki=0.001) with a tight ±8000-tick integrator clamp and back-calculation anti-windup. It is parked because the inner HWZC timing PI and this outer loop form a nested loop that cascades to PSU-OV → phantom 200 k → UV above ~80 k; revival needs an absolute output slew limiter first. Its telemetry (speedPiTarget/LastError/OutputDuty/IntegratorF) is still plumbed.

10  GSP Host Protocol & Parameter System

The Garuda Serial Protocol is a binary command/telemetry API over UART1 at 115.2 kbps, polled (no UART ISR), pumped from main() via GSP_Service() (PumpRx → ParserProcess → TelemTick → PumpTx). It is the host-facing API for the GUI and MCP tooling.

10.1  Wire protocol & CRC

Versioning is pull-based: the host issues GET_INFO (0x01) and reads protocolVersion (GSP_PROTOCOL_VERSION=3) from the 24-byte INFO frame, then decodes everything else accordingly. Snapshot fields are append-only so older decoders parse the prefix and ignore the tail. On CRC mismatch the frame is silently dropped; the host retries.

13  Worked Example — One Commutation Cycle at 100 000 eRPM

To make the timing chain concrete, trace a single sector with the reference profile (45 kHz PWM, maxClosedLoopErpm=240 000, timingAdvMaxDeg=20°, RAMP_TARGET_ERPM=3000) running steadily at 100 000 eRPM, duty ≈ 38 %, Vbus ≈ 10 V.

Step A — period in HR ticks

stepPeriodHR = 109/eRPM = 109/100 000 = 10 000 ticks = 100 µs per 60° sector
one electrical revolution = 6 × 100 µs = 600 µs ⇒ 1 667 erev/s

At 45 kHz the ADC ISR runs every 22.22 µs, so a 100 µs sector spans only ≈ 4.5 ADC samples — already coarse for the software path, which is why HWZC (1 MHz comparator) owns detection here.

Step B — advance angle

advDeg = 0 + (20−0)·(100 000 − 3 000)/(240 000 − 3 000) = 20·(97 000/237 000) = 8.18° → 8°
advFp8 = ((30+8)·256)/60 = 9728/60 = 162  (Q8, = 0.633)

Step C — setpoint & commutation instant

setValue = (advFp8·T)≫8 = (162·10 000)/256 = 6328 ticks (= 63.3 % into the sector)
reactive commDelay = T·(30−advDeg)/60 = 10 000·(22/60) = 3667 ticks after the ZC

The ZC ideally lands at the sector midpoint (5000 ticks / 30°). With 8° advance the loop holds the ZC at 6328 ticks past the previous commutation, so commutation fires 8° before the next rotor sector boundary — covering the inductive + detection lag.

Step D — filter-lag compensation bias

omegaTauQ15 = K/stepP = 102 943 706 / 10 000 = 10 294 (= ωeτ = 0.314 rad = 18°)
ωe = 2π·(100 000/60) = 10 472 rad/s ⇒ ωeτ = 10 472·30µs = 0.314 ✓ (matches)
assume amp (zcAmpForFilterComp) ≈ 220 counts ⇒ offset = (220·10 294 ≫ 15)·50/100 = (69)·0.5 = 35 counts
rising CMPLO = zcThreshold − 35 + 4(deadband) ; falling CMPLO = zcThreshold + 35 − 4

So at 100 k the comparator threshold is pulled ≈ 35 counts toward the oncoming signal to cancel an 18° RC lag — the single largest dynamic correction in the loop, and the reason ZC stays centred rather than walking late with speed.

Step E — blanking & plausibility window

blankTicks = 10 000·14/100 = 1400 ticks (14 µs) masked after each commutation
plausibility floor = max(10 000·50/100, RT_HWZC_MIN_STEP_TICKS=4167) = 5000 ticks — any interval < 5000 rejected

Step F — PI correction for a 200-tick early capture

say capValue = 6128 (200 ticks early) ⇒ delta = 6128 − 6328 = −200
clamp (mid RPM, T > 6667-tick HIGH threshold ⇒ symmetric ±T/8 = ±1250): −200 passes
integratorF += 0.0625·(−200) = −12.5 ⇒ Tnew = integratorF + 0.25·(−200) = T − 12.5 − 50 ≈ 9938 ticks
new commanded eRPM = 109/9938 = 100 624 — a 0.6 % nudge faster to recentre the ZC

The tight negative clamp (here T/8, tightening to T/32 above 150 k) prevents a string of early captures from ratcheting the period down onto a half-period phantom; the ABS_FLOOR guard (§8) is the absolute backstop if it ever tried.

The single most important invariant: the firmware never commands the rotor faster than it can physically be — not via the PI (asymmetric clamp), not via harmonic lock (ABS_FLOOR), not via the speed map (period floor). 18.1  Bringing up a new motor — checklist

  1. λ first. Set focKeUvSRad = round(5.5132×106/(KV·PP)). A wrong λ pins the motor at a clean fraction of its real speed (§8) — diagnose this before touching anything else.
  2. Pole pairs. motorPolePairs = magnet poles ÷ 2.
  3. Speed cap. maxClosedLoopErpm = nominal no-load (KV·V·PP) is a safe default; this is also the advance anchor and the period floor. Live-tune 0x11 upward to explore field-… (commutation-advance)-limited headroom.
  4. Startup. Port a proven low-hand-off startup (rampTarget ≈ 3000, align/sineMod low) before tuning the run loop.
  5. OC chain. Scale ocLimitMa/ocSwLimitMa/ocFaultMa to the motor's stall current — the 24 V-class thresholds will never protect a 7 A micro.
  6. Crossover & falling cap. High-KV/low-L motors: raise hwzcCrossoverErpm (engage HWZC only where BEMF is solid) and lower HWZC_FALLING_SW_MAX_ERPM (coast falling before the mid-band oscillation builds).
